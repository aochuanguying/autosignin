# 旁路由 VLESS 入站端口改动复盘

## 背景

旁路由 (192.168.50.2) 原有一个 VLESS 入站端口 16825，用于从外网（手机 4G）通过 Shadowrocket 连回家庭内网。因为主路由已将 16825 端口映射到 NAS（192.168.50.50），需要把旁路由的 VLESS 入站改为 16824 端口。

## 端口用途说明

| 端口 | 映射目标 | 用途 |
|------|----------|------|
| 16825 | 192.168.50.50 (NAS) | NAS 的 VLESS 服务（访问家庭内网文件） |
| 16824 | 192.168.50.2 (旁路由) | 旁路由的 VLESS 入站（翻墙+内网访问） |

## 最终配置状态

### 旁路由 `/usr/local/xray/config.json` 中的 vless-in

```json
{
  "tag": "vless-in",
  "port": 16824,
  "listen": "0.0.0.0",
  "protocol": "vless",
  "settings": {
    "clients": [
      {
        "id": "219eb005-4294-4e78-5514-088d907657e6",
        "level": 0
      }
    ],
    "decryption": "none"
  },
  "sniffing": {
    "enabled": true,
    "destOverride": ["http", "tls"]
  }
}
```

- 纯 VLESS TCP，无 Reality，无 flow
- 原因：Reality 在当前网络环境下无法正常工作（见下方分析）

  "settings": {
    "vnext": [{
      "address": "www.hxfssc.com",

- 同样纯 VLESS TCP，无 Reality，无 flow，与旁路由配对

### Shadowrocket 分享链接

```

```

## 旁路由 nftables 改动

### `/usr/local/xray/tproxy.nft` 新增规则

相比原始配置，增加了两条规则排除 16824 端口流量：

1. **prerouting chain**: `tcp dport 16824 return`
   - 作用：从外网进来的目标为 16824 的包不走 tproxy，直接送到 xray 的 vless-in 监听
   - 原因：没有这条规则，外网来的 16824 流量会被兜底规则 tproxy 到 12345 端口

2. **output chain**: `tcp sport 16824 return`
   - 作用：xray vless-in 的回包（源端口 16824）不被打 fwmark 1
   - 原因：没有这条规则，SYN-ACK 等回包会被标记 fwmark 1，走 table 100 路由到 lo，外部客户端收不到回复

### `inet filter input` chain 新增规则

```
tcp dport 16824 accept
```

- 作用：防火墙放行外网到 16824 端口的入站连接
- 原因：input chain 默认策略是 DROP，只放行了特定端口。之前只放行了 16825，改端口后必须加 16824
- **这是导致手机长时间连不上的直接原因**

## 为什么 Reality 不可用

### 尝试过程

1. 最初配置了 VLESS + Reality（dest: www.microsoft.com:443）
2. 手机连接时 TCP 三次握手成功，但 TLS/Reality 握手超时
3. 通过 `show: true` 调试发现：Reality 服务端能收到客户端 ClientHello，也能从 dest 获取部分 TLS 数据（Server Hello、Certificate），但 `isHandshakeComplete: false`

### 根本原因

旁路由到 www.microsoft.com:443 的 TLS 握手延迟 **~1200ms**，Reality 需要完整获取 dest 的 TLS 握手数据后才能回复客户端。在这 1-2 秒的等待中，Shadowrocket 客户端已经超时断开。

### 尝试的 dest 目标

| dest | 延迟 | 结果 |
|------|------|------|
| www.microsoft.com:443 | ~1200ms | 超时 |
| www.apple.com:443 | ~120ms | xray 警告会被 GFW 封锁 |
| itunes.apple.com:443 | ~100ms | xray 警告会被 GFW 封锁 |

### 结论

- Reality 设计用于翻墙出口伪装（比如 proxy 出口的 154.26.187.89:443 用 Reality 伪装）
- 旁路由 16824 是**入站服务**（从外网连回家），不需要伪装
- 如果运营商真的干扰纯 VLESS，可以考虑在 vless-in 上加普通 TLS（自签证书或 Let's Encrypt），而非 Reality

## 频繁出现 nftables/ip rule 丢失的原因

### 根本原因

旁路由没有使用 `/etc/init.d/xray` 服务管理脚本来管理 xray 进程。当前 xray 是**手动 nohup 启动的裸进程**，不与 nftables 规则和 ip rule 绑定。

### 问题链

1. 每次 `kill` xray 进程时，SSH 连接经过 tproxy → xray 处理
2. xray 被杀后，tproxy 规则把 SSH 流量送到已经不存在的 12345 端口，SSH 立即断开
3. SSH 断开意味着后续的 `nohup xray run ...` 命令没有执行
4. 新的 xray 进程没启动 → nftables 规则把所有流量送到空端口 → 整台机器不可达
5. 需要通过 socks 代理（本机 xray → 旁路由 16824 vless-in）或等待重连才能恢复

### 为什么 nftables 规则会"丢失"

实际上规则没有丢失。但在某些重启过程中：
- `nft delete table ip xray` 删除了规则后 xray 没启动成功
- 或者 xray 启动了但 `nft -f` 命令没执行（SSH 断了）

### 正确的重启方式（避免 SSH 断开）

```bash
# 方法一：一条命令完成（原子操作）
kill $(pgrep -f "xray run"); sleep 1; nohup /usr/local/xray/xray run -c /usr/local/xray/config.json > /tmp/xray.log 2>&1 &

# 方法二：使用 systemd service（推荐）
systemctl restart xray
```

### 推荐修复：创建 systemd service

应该创建一个 systemd service 来管理 xray，在 `ExecStartPre` 中设置 ip rule/nftables，在 `ExecStopPost` 中清理：

```ini
[Unit]
Description=Xray TPROXY Service
After=network.target

[Service]
Type=simple
ExecStartPre=/sbin/ip route add local default dev lo table 100
ExecStartPre=/sbin/ip rule add fwmark 1 table 100
ExecStartPre=/usr/sbin/nft -f /usr/local/xray/tproxy.nft
ExecStart=/usr/local/xray/xray run -c /usr/local/xray/config.json
ExecStopPost=/usr/sbin/nft delete table ip xray
ExecStopPost=/sbin/ip rule del fwmark 1 table 100
ExecStopPost=/sbin/ip route del local default dev lo table 100
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

## 待解决问题

1. **Shadowrocket 连接 16824 后无法翻墙**
   - 16:27 的日志显示手机流量确实通过 vless-in 进入，并正确路由到 proxy（翻墙）出口
   - 后续连接失败可能是运营商 DPI 识别裸 VLESS 协议后封锁了端口
   - 建议：给 vless-in 添加 TLS 加密（自签证书），避免明文 VLESS 被 DPI 识别

2. **inet filter input 规则持久化**
   - 当前 `tcp dport 16824 accept` 规则是临时添加的（重启会丢失）
   - 需要写入防火墙配置文件使其开机自动加载

3. **ip rule 重复**
   - 多次执行 `ip rule add fwmark 1 table 100` 导致规则重复
   - 添加前应先检查是否已存在
