# 旁路由内存泄漏排查与修复（2026-07-24）

## 问题现象

倍控 G31 旁路由（Debian 13, 4GB DDR4）内存占用持续升高，Xray 进程 RSS 达到 2.7GB，几乎吃满物理内存。

## 根因分析

### 路由回环（Routing Loop）

公司服务器 `103.94.200.246`（内网 IP `10.30.5.33`）上运行的 Xray 配置存在缺陷：

1. 服务器 Xray 的 `officebound` 路由规则 IP 列表缺少 `10.16.0.0/16` 和 `10.33.0.0/16`
2. 到这两个网段的流量被兜底规则 `nasbound` 发送到旁路由的 `vless-in`（端口 16825）
3. 旁路由收到后，routing 匹配 `10.16.0.0/16 → office` 出口，又发回公司服务器
4. 公司服务器收到后又兜底发回旁路由 → 无限回环

### 回环链路图

```
公司服务器 (10.30.5.33 / 103.94.200.246) Xray
  → 请求目标 10.16.157.144:7000
  → officebound 规则不匹配（缺少 10.16.0.0/16）
  → 兜底 nasbound → www.hxfssc.com:16825（旁路由 vless-in）

旁路由 Xray vless-in 收到请求
  → routing: 10.16.0.0/16 → office 出口
  → office 出口连接 103.94.200.246:443 (xhttp)

公司服务器再次收到 → 兜底 nasbound → 旁路由
  → ∞ 无限回环（每秒 10+ 新连接）
  → goroutine + TCP buffer 持续增长 → 内存耗尽
```

### 次要问题

- Xray 日志级别被设置为 `debug`（应为 `warning`），产生大量磁盘 IO（error.log 达 1.2GB/天）

## 修复措施

### 1. 公司服务器 Xray 配置（根治）

**文件**：`10.30.5.33:/usr/local/etc/xray/config.json`

删除 `nasbound` 出口（不再有流量发回旁路由），所有入站流量统一走 `officebound`（通过 `10.19.34.236:9087` 中转访问内网资源）。

修改前：
```json
"routing": {
  "rules": [
    { "domain": ["domain:hisense.com"], "outboundTag": "officebound" },
    { "ip": ["10.19.0.0/16", "172.16.0.0/16", "10.30.0.0/16"], "outboundTag": "officebound" },
    { "network": "tcp,udp", "outboundTag": "nasbound" }  // ← 兜底发到旁路由，回环根源
  ]
}
```

修改后：
```json
"routing": {
  "rules": [
    { "network": "tcp,udp", "outboundTag": "officebound" }  // 所有流量走 officebound
  ]
}
// nasbound 出口已删除
```

### 2. 旁路由 Xray 配置（防御性修复）

**文件**：`192.168.50.2:/usr/local/xray/config.json`

新增 routing rule（插入到最前面）：`vless-in` 入站 + 目标为公司内网 IP → `direct`。即使公司服务器配置有误，也不会形成回环。

```json
{
  "type": "field",
  "inboundTag": ["vless-in"],
  "ip": ["10.19.0.0/16", "172.16.0.0/16", "10.30.0.0/16", "10.33.0.0/16", "10.16.0.0/16"],
  "outboundTag": "direct"
}
```

### 3. 旁路由 nftables 规则修复

**文件**：`192.168.50.2:/usr/local/xray/tproxy.nft`

将 `meta mark 0x000000ff return`（Xray 自身流量排除）移到 `office_nets` 匹配规则**之前**。

修改前（output chain）：
```nft
ip daddr @office_nets meta l4proto { tcp, udp } meta mark set 0x00000001 accept
meta mark 0x000000ff return  // ← 晚了，mark 已被覆盖
```

修改后：
```nft
meta mark 0x000000ff return  // ← 先排除 Xray 自身流量
tcp sport 16825 return       // ← VLESS 入站回包排除
ip daddr @office_nets meta l4proto { tcp, udp } meta mark set 0x00000001 accept
```

### 4. update-office-nets.sh 同步逻辑更新

**文件**：`192.168.50.2:/usr/local/xray/update-office-nets.sh`

原脚本只同步 `outboundTag == "office"` 的 IP 规则。新增了对 vless-in 防回环规则的同步，确保通过面板修改内网 IP 段时两处保持一致。

新增的 jq 匹配条件：
```
elif .outboundTag == "direct" and .inboundTag == ["vless-in"] and .ip != null then .ip = $ips
```

### 5. 日志级别修正

`loglevel` 从 `debug` 改回 `warning`。

## 修复效果

| 指标 | 修复前 | 修复后 |
|------|--------|--------|
| 内存占用 | 3.2Gi / 3.6Gi (89%) | 590Mi / 3.6Gi (16%) |
| Xray RSS | 2.7GB | 85MB |
| 到 office 连接数 | 3686+ 且持续增长 | 2~3 稳定 |
| 可用内存 | 400Mi | 3.0Gi |

## 经验教训

1. **路由兜底规则要谨慎**：公司服务器的 `nasbound` 兜底规则把未匹配的流量发到旁路由，本意是让旁路由代理，但旁路由又把公司内网 IP 发回来，形成回环。
2. **防御性路由设计**：在旁路由 vless-in 入站加一条防回环规则（公司内网 IP → direct），即使上游配置有误也不会回环。
3. **nftables 规则顺序很重要**：mark 255 排除必须在 office_nets 匹配之前，否则 Xray direct 出口的流量会被重新 tproxy。
4. **统一维护要覆盖所有消费者**：`update-office-nets.sh` 需要同步更新所有引用内网 IP 段的规则，包括新增的防回环规则。
